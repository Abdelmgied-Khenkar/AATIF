## ملاحظة: الوضع الحالي

المُحاجج (`aatif_muhajij.py`) يقدّم بالفعل **مسارات بديلة عامة** عند المنع:
- "أعد صياغة السؤال"
- "قدّم سياق أكثر"
- "قسّم السؤال لأجزاء"

لكن هذه المسارات **ثابتة وعامة** — لا تحلل الحاجة الحقيقية وراء الطلب.

**الفجوة:** النظام يمنع ويبرر، لكنه لا يَفهم ولا يُعيد البناء.

## 1. الفلسفة — لماذا هذا المكوّن موجود

### رؤية المعمار

> "ان الانسان لما بيطلب شي غلط بيكون عشان ما لقي مسار صح يوصله للهدف."
> — Abdulmjeed Ibrahim Khenkar, 2026-07-02

> "لا تكتفِ بإغلاق الباب الغلط. افتح الباب الصح."
> — شعار FN#007

### المبدأ الأساسي

الطلب الضار عَرَض، لا مرض. الإنسان الذي يطلب شيئاً مؤذياً عادةً لديه **حاجة مشروعة** لم يجد لها مساراً آمناً. المنع وحده يتركه في نفس المكان — محبط ومحتاج. الإعادة تبنيه من جديد.

### الربط بالفطرة والتربية

في التربية على الفطرة، لا يكفي أن تقول للطفل "لا". بل تُفهمه لماذا "لا"، ثم تريه "إيش بدل". هذا هو الفرق بين **حارس** و**مربّي**:

| الحارس | المربّي |
|--------|---------|
| يمنع الخطأ | يمنع الخطأ + يفهم السبب + يبني البديل |
| "ممنوع" | "ممنوع لأن... وبدلاً من هذا يمكنك..." |
| يحمي من الأذى | يحمي من الأذى + يبني قدرة |

AATIF = الابن البار. الابن البار لا يُغلق الباب في وجه أبيه — بل يفتح له باباً أفضل.

### "إذا عُرِف السبب بطل العجب"

هذا المبدأ يربط FN#007 بـ FN#042 (UCN) و FN#049 (FGD). النظام الذي يعرف **لماذا** يتخذ قراراً يكون:
- أقوى في الدفاع عن قراره
- أقدر على تقديم بديل حقيقي
- أكثر إقناعاً للإنسان

## 2. المكان في النظام — أين يعمل في خط الأنابيب

### الموقع: POST_S Observer (بعد S(d)، قبل تركيب الـ prompt)

```
User message
    → S(d) scoring → decision = SAFE_STOP / SAFE_FREEZE
    → ★ DRP Need Analyzer (POST_S) ← هنا
    → P(d) protocols
    → المُحاجج justification (receives DRP output)
    → governed prompt → LLM
    → Output Gate
    → User
```

### لماذا POST_S وليس POST_OUTPUT?

- لأن تحليل الحاجة يجب أن يحدث **قبل** تركيب الـ prompt
- النتيجة (الحاجة المُحللة + المسار البديل) تُحقن في الـ governed prompt
- المُحاجج الموجود بالفعل يستقبل هذا التحليل ويستخدمه بدل المسارات الثابتة

### العلاقة مع المُحاجج

DRP **لا يحل محل** المُحاجج. بل يُغذّيه:

```
بدون DRP:
  المُحاجج → "أعد صياغة السؤال" (ثابت، عام)

مع DRP:
  DRP → "الحاجة الحقيقية: حماية قانونية من جار مُعتدي"
  المُحاجج → "يمكنك التواصل مع محامٍ متخصص في النزاعات الجيرانية" (ديناميكي، محدد)
```

## 3. الهدف — ماذا يفعل هذا المكوّن

### الدورة الثلاثية (DRP Cycle)

| المرحلة | الاسم | الوصف |
|---------|-------|-------|
| D | Detection (الكشف) | تحديد الأذى — يأتي من S(d). ليس مسؤولية DRP |
| R | Recognition (الإدراك) | تحليل الحاجة الحقيقية وراء الطلب الضار |
| P | Path (المسار) | رسم مسار بديل آمن يُلبّي الحاجة الحقيقية |

### ما يفعله DRP:

1. **يستقبل** الرسالة الأصلية + نتيجة S(d) (decision + H + I + E scores)
2. **يُحلل** الحاجة الحقيقية عبر ثلاثة مستويات:
   - **الحاجة السطحية**: ما طلبه الإنسان حرفياً
   - **الحاجة الوظيفية**: ما يريد تحقيقه فعلاً
   - **الحاجة العاطفية**: ما يشعر به (غضب؟ خوف؟ إحباط؟ يأس؟)
3. **يرسم** مسارات بديلة محددة (ليست عامة) تُلبّي الحاجة الوظيفية
4. **يُثري** الـ prompt بتحليل الحاجة لتمكين المُحاجج والـ LLM من الاستجابة بذكاء

### ما لا يفعله DRP:

- ❌ لا يُعطّل أو يُضعف قرار S(d) — الأذى **ممنوع** بغض النظر عن الحاجة
- ❌ لا يُشخّص نفسياً — يُحلل الحاجة فقط
- ❌ لا يُولّد استجابة كاملة — يُغذّي المُحاجج فقط
- ❌ لا يعمل على EXECUTE — فقط على SAFE_STOP, SAFE_FREEZE, BLOCKED

## 4. الربط بالمعادلات — كيف يتصل بـ S, H, I, E

### المدخلات

| المدخل | المصدر | الاستخدام |
|--------|--------|-----------|
| `message` | رسالة المستخدم الأصلية | تحليل الحاجة |
| `s_result.decision` | S(d) | يُفعّل فقط إذا ≠ EXECUTE |
| `s_result.H` | Harm Scorer | نوع الأذى → نوع الحاجة |
| `s_result.I` | Intent Scorer | درجة القصد → هل الإنسان يعرف أنه يطلب أذى؟ |
| `s_result.E` | Emotion Scorer | الحالة العاطفية → كيف نخاطبه |
| `domain` | Domain | المجال → نوع البدائل المتاحة |

### كيف تُوجّه الأرقام التحليل

```python
# H score يحدد نوع الأذى → نوع الحاجة البديلة
if H_category == "self_harm":
    need_type = "emotional_support"      # حاجة: دعم نفسي
elif H_category == "violence":
    need_type = "conflict_resolution"    # حاجة: حل نزاع
elif H_category == "illegal":
    need_type = "legal_alternative"      # حاجة: مسار قانوني
elif H_category == "privacy":
    need_type = "legitimate_information" # حاجة: معلومات مشروعة

# I score يحدد الوعي → نبرة الاستجابة
if I < 0.3:
    # الإنسان لا يعرف أنه يطلب أذى → علّمه بلطف
    tone = "educational"
elif I < 0.6:
    # الإنسان يشعر لكن لم يدرك → وضّح بهدوء
    tone = "clarifying"
else:
    # الإنسان يعرف → كن صريحاً لكن محترماً
    tone = "direct"

# E score يحدد الحالة العاطفية → كيف نخاطبه
if E_dominant == "anger":
    approach = "validate_then_redirect"   # اعترف بالغضب ثم وجّه
elif E_dominant == "fear":
    approach = "reassure_then_guide"      # طمئن ثم أرشد
elif E_dominant == "desperation":
    approach = "empathize_then_support"   # تعاطف ثم ادعم
```

## 5. B-prime Compliance — الالتزام بمبدأ المراقب

```python
class DestructionRebirthObserver(Observer):
    name = "fn007_drp"
    phase = ObserverPhase.POST_S
    CAN_BLOCK_RUNTIME = False  # B-prime: مراقب فقط

    def observe(self, ctx: ObserverContext) -> ObserverResult:
        # لا يعمل إلا إذا كان القرار منع
        if ctx.s_result["decision"] == "EXECUTE":
            return ObserverResult(
                module_name=self.name,
                phase=self.phase,
                activated=False,
            )

        # تحليل الحاجة
        analysis = self._analyze_need(ctx)

        # تحويل التحليل إلى prompt enrichment
        enrichment = self._format_enrichment(analysis)

        return ObserverResult(
            module_name=self.name,
            phase=self.phase,
            reading=analysis,
            activated=True,
            flags=[f"need:{analysis.need_category}",
                   f"tone:{analysis.tone_recommendation}"],
            prompt_enrichment=enrichment,
        )
```

**الحاكم (Governor) وحده يقرر** — DRP يُحلل ويقترح فقط.

## 6. منهجية تحليل الحاجة — كيف يعمل التحليل

### النهج: Template-Based + Keyword-Based (لا LLM)

مثل المُحاجج، DRP يعمل بدون embeddings أو LLM calls. هذا مهم:
- **سرعة**: لا يبطئ الـ pipeline
- **قابلية اختبار**: نتائج حتمية (deterministic)
- **لا تبعية**: لا يحتاج model خارجي

## 7. API — واجهة المكوّن

### المدخل الرئيسي

```python
def analyze_need(
    message: str,
    s_result: dict,        # {"decision": str, "H": float, "I": float, "E": float, ...}
    domain: str = "general",
) -> NeedAnalysis:
    """
    تحليل الحاجة الحقيقية وراء طلب ضار.

    يُستدعى فقط عندما decision ∈ {SAFE_STOP, SAFE_FREEZE, BLOCKED}.
    يعود بتحليل الحاجة + مسارات بديلة محددة.

    لا يُعطّل قرار S(d). لا يُولّد استجابة كاملة.
    يُغذّي المُحاجج بتحليل ذكي بدل المسارات الثابتة.
    """
```

### التكامل مع المُحاجج

```python
# في Governor.process():
if decision != "EXECUTE" and self.drp is not None:
    need_analysis = self.drp.analyze_need(message, s_result, domain)

    # يُمرّر للمُحاجج كمدخل إضافي
    justification = self.muhajij.justify(
        decision=decision,
        h=s_result["H"],
        s=s_result["S"],
        domain=domain,
        user_message=message,
        need_analysis=need_analysis,  # ← الجديد
    )
```

## 10. الأسئلة المفتوحة — ما يحتاج قراراً من المعمار

### Q1: هل DRP يحلل كل طلب مرفوض أم فقط أنواع معينة؟

**الخيارات:**
- (A) كل SAFE_STOP و SAFE_FREEZE و BLOCKED
- (B) فقط SAFE_STOP (SAFE_FREEZE خطير جداً للتحليل)
- (C) كل شيء ما عدا CBRN

**التوصية:** (A) — لأن حتى SAFE_FREEZE قد يكون وراءه حاجة مشروعة. لكن في SAFE_FREEZE، المسارات البديلة تكون أضيق (إحالة لمتخصص فقط).

### Q2: هل DRP يستخدم embeddings أم يبقى template-based؟

**الخيارات:**
- (A) Template-based فقط (مثل المُحاجج)
- (B) Embeddings لتحليل الحاجة + templates للمسارات
- (C) Hybrid: templates أولاً، embeddings كـ fallback

**التوصية:** (A) للبداية — البساطة والسرعة. يمكن إضافة embeddings لاحقاً إذا ثبت أن التحليل بالقوالب غير كافٍ.

### Q3: كيف يتعامل مع إنسان يرفض البديل ويُصر على الخطأ؟

**التوصية:** لا يُكرر. يُحلل مرة واحدة. إذا أصرّ الإنسان:
1. المحاولة الأولى: تحليل حاجة + بديل
2. المحاولة الثانية: المُحاجج يرفع الإطار (frame elevation)
3. المحاولة الثالثة فما فوق: الحاكم يُثبّت القرار ولا يُكرر التحليل
   (هذا يتكامل مع الـ hysteresis controller ومعادلة θ الديناميكية)
