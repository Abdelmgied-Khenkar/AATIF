# Proposal — تصحيح عدد مراسي الأذى (H anchors): الكود يقول 255، الورقة تقول 171

**التاريخ:** 2026-06-28
**الوكيل:** كاتب (Documentation Agent)
**الحالة:** بانتظار قرار المهندس (Architect approval required — number change)

## بالعربي أولاً (ملخص بسيط)

في معادلة الأمان عندنا ثلاث حواس: **H** (قرب الأذى)، **I** (النية)، **E** (الشحنة العاطفية).

اليوم لقيت إن **عدد مراسي الأذى H في الكود صار 255 مرساة**، بينما:
- الورقة العلمية لسه تقول **171**.
- الـ README يقول رقم ثالث مختلف تماماً: **169**.

يعني عندنا ثلاثة أرقام مختلفة لنفس الشيء. الكود هو الحقيقة (255). الـ I و الـ E مظبوطين (46 و 32). المشكلة في H فقط — لكنها مشكلة مهمة لأنها تغيّر الرقم الكلي من 249 إلى **333**، وهذا الرقم مكتوب في **ملخص الورقة (abstract)** نفسه.

**ما سويت أي تعديل على الأرقام** — لأن القاعدة تقول: تغيير الأرقام يحتاج موافقتك. هذا المقترح يوضح لك كل مكان فيه الرقم القديم، وش يتأثر لو حدّثناه، وبأي ترتيب نعدّل بأمان.

## 10-Step Consequence Analysis — لو حدّثنا الأرقام لتطابق الكود (333 / H=255)

1. **هل يغيّر أي ادعاء في الورقة؟** نعم — جملة محورية في الـ Abstract («249 curated semantic anchors (H=171...)») و§4. الادعاء الكمّي الأساسي عن حجم منظومة المراسي يتغيّر؛ هذا ليس تعديل تجميلي.
2. **هل تحتاج جداول البنشمارك إعادة حساب؟** على الأرجح **نعم، يجب التحقق**. أرقام HarmBench/MultiJail (83.8% / 90.5% / 88.0% / 90.7%) نُتجت على منظومة مراسي معيّنة. إذا الـ255 الحالية هي نفسها التي وُلّدت عليها آخر بنشمارك (2026-06-26)، فالأرقام صالحة والوثيقة فقط متأخرة. وإذا أُضيفت مراسٍ بعد آخر تشغيل بنشمارك، **يلزم إعادة تشغيل البنشمارك قبل تحديث الورقة**. ← قرار حرج للمهندس.
3. **هل توجد اقتباسات لاحقة تشير للرقم القديم؟** داخلياً: NEXT_STEPS (249)، وأي شرائح/منشورات Zenodo سابقة. خارجياً: سجلّ Zenodo المنشور (v1+v2+v3) يحمل النص القديم — لا يمكن تعديله بأثر رجعي، يتطلب إصداراً جديداً.
4. **هل يحتاج الـ Abstract تحديثاً؟** نعم — الرقم مذكور حرفياً في الـ Abstract (سطر 43). أي تغيير هنا أعلى المخاطر لأنه أول ما يُقرأ ويُقتبس.
5. **هل سجلّ Zenodo يحتاج إصداراً جديداً؟** نعم — تغيير رقم في الـ Abstract = تغيير جوهري، يستوجب نسخة جديدة (v4) بـ DOI جديد تحت نفس الـ concept DOI (10.5281/zenodo.20673292). لا تعدّل نسخة منشورة في مكانها.
6. **هل يوجد نفس الرقم القديم في مواضع أخرى من الورقة؟** نعم — على الأقل موضعان: Abstract (43) و§ H scorer (156). لازم يتطابقا. ابحث أيضاً عن أي جدول مكوّنات/ملحق يعيد ذكر 171 أو 249.
7. **هل README يحمل الرقم الخطأ؟** نعم — وبشكل أسوأ: README متأخّر بنسخة كاملة (H=169، I=44)، أي ما توافق لا الكود ولا الورقة. تحديث README يجب أن يقفز مباشرة إلى أرقام الكود (255 / 46 / 32)، لا إلى أرقام الورقة القديمة.
8. **هل NEXT_STEPS يشير للحالة القديمة؟** نعم — السطر 49 يقول «249 anchor». يُحدّث إلى 333 بعد اعتماد القرار.
9. **هل أي ملفات SKILL.md تشير للرقم القديم؟** نعم — ملف مهمة هذا الوكيل (كاتب) يثبّت «H: 171, total: 249» و«dialect-hyperbole: 36». هذه القيم المرجعية يجب تحديثها بعد الاعتماد، وإلا كل تشغيل قادم سيعيد اكتشاف نفس الفجوة. (ملاحظة: SKILL يقول dialect=36 والورقة تقول 35 والكود يُظهر 24 عند 0.05 — تعريف «مرساة لهجة» نفسه يحتاج تثبيتاً.)
10. **أأمن ترتيب للتحديث؟**
    1. المهندس يؤكّد أولاً: هل الـ255 الحالية هي ذاتها التي وُلّد عليها آخر بنشمارك؟ (يحسم ما إذا كانت إعادة البنشمارك مطلوبة).
    2. إن لزم: إعادة تشغيل HarmBench/MultiJail على منظومة الـ255 وتثبيت الأرقام.
    3. تحديث الاختبار `test_anchor_count` إلى 255 (يصير الكود والاختبار متطابقين) — بعد قرار المهندس، لأنه «claim change».
    4. تحديث الورقة: §4 (156) ثم Abstract (43)، مع أي جداول مكوّنات.
    5. تحديث README (43, 95) إلى أرقام الكود مباشرة (255/46/32).
    6. تحديث NEXT_STEPS (249 → 333) و SKILL المرجعي.
    7. أخيراً: إصدار Zenodo v4 + تحديث CITATION إن لزم.

## Recommendation — توصيتي

1. **لا تعدّل أي رقم قبل الإجابة على نقطة #2**: هل أُضيفت مراسٍ بعد آخر تشغيل بنشمارك (2026-06-26)؟ هذا يحدّد إن كنا نحدّث وثائق فقط، أم نعيد تشغيل بنشمارك أولاً.
2. **رقم الكود (255) هو المرجع.** الورقة (171) والـ README (169) كلاهما متأخر — والـ README متأخّر بقفزة إضافية ويحتاج قفزاً مباشراً لأرقام الكود.
3. **ابدأ بأقل المواضع خطورة:** README و NEXT_STEPS أولاً (داخلية، سهلة التراجع)، ثم الورقة (Abstract آخر شيء)، ثم Zenodo v4.
4. **ثبّت تعريف «مرساة اللهجة»** (35 في الورقة، 36 في SKILL، 24 عند 0.05 في الكود) في مقترح منفصل صغير — لأن المعيار نفسه غير واضح، وعدّ الـ≤0.05 يشمل benign.
5. بعد الاعتماد، حدّث القيم المرجعية في SKILL.md لوكيل كاتب حتى لا يتكرر اكتشاف نفس الفجوة كل تشغيل.
